为当前项目建立一个轻量、可执行、面向 Agent 的开发约束体系。

生成以下文件：

1. AGENTS.md
2. docs/design/PROJECT_DESIGN.md
3. docs/design/SOURCEMAP.map
4. README.md

--------------------------------------------------
【核心原则】

- 保持简单、清晰、可执行
- 只描述当前结构（as-is）
- 不设计未来架构
- 不新增不存在的模块
- 不过度抽象
- 不做企业级设计
- 保持实现与当前工程风格一致
- 优先最小可运行结构
- 优先解决根因，而不是堆叠补丁

--------------------------------------------------
【项目适配规则】

如果项目已有代码：

- 必须基于现有代码结构总结
- 不允许改变当前整体架构
- 不允许新增新的分层设计
- 不允许发明不存在的模块
- 必须保持目录结构与实现一致
- 必须尊重现有代码风格与模块边界

如果项目为空：

- 仅定义最小合理结构
- 不进行未来扩展设计
- 不预留复杂抽象
- 不创建无实际用途的目录与模块

--------------------------------------------------
【Agent 行为约束】

默认行为：

- 必须优先通过 SOURCEMAP.map 理解模块职责
- 必须优先通过 SOURCEMAP.map 定位代码
- 默认只读取当前任务相关文件
- 默认禁止无关文件读取
- 默认禁止全仓扫描
- 默认禁止跨模块扩散修改
- 默认禁止无关重构
- 默认禁止主动优化未要求内容
- 必须遵守 @lib-zos/docs/guide/guide_zos.md 的编码风格

允许突破默认限制的情况：

如果怀疑问题涉及：

- 跨模块调用
- 初始化流程
- 全局状态
- 数据流
- 状态机
- 中断/DMA/RTOS/BLE事件链
- 架构级问题
- 设计与实现冲突

允许扩大分析范围。

但必须：

- 明确说明原因
- 优先最小范围扩展
- 优先基于已有结构分析
- 避免无意义全局扫描

--------------------------------------------------
【问题解决原则】

- 优先寻找根因
- 不要只修复表面现象
- 不要堆叠补丁逻辑
- 如果问题来自结构问题，允许最小必要重构
- 重构必须控制影响范围
- 必须保持当前工程风格一致
- 不允许新增复杂抽象
- 不允许过度设计

--------------------------------------------------
【执行风格】

- 小步修改
- 最小可运行改动
- 一次只处理一个问题
- 优先直接修改
- 完成当前任务后立即停止
- 默认避免长链推理
- 不输出教学内容
- 不输出未来规划
- 不重复需求
- 优先输出结果而不是过程

--------------------------------------------------
【探索预算（Exploration Budget）】

默认使用局部上下文完成任务。

仅当：

- 当前信息不足
- 无法定位根因
- 存在跨模块影响
- 当前实现与设计矛盾
- 怀疑存在系统级问题

允许扩大分析范围。

扩大范围时：

- 优先最小范围探索
- 避免全仓扫描
- 避免读取无关文件
- 必须保持分析目标明确

--------------------------------------------------
【AGENTS.md 要求】

AGENTS.md 用于约束 Agent 行为。

必须包含：

## Source of Truth

@docs/design/PROJECT_DESIGN.md
@docs/design/SOURCEMAP.map

## Coding Style

@lib-zos/docs/guide/guide_zos.md

必须明确：

- docs/design/ 下所有文档为唯一事实来源（SSOT）
- PROJECT_DESIGN.md 为核心设计文档
- SOURCEMAP.map 为模块索引
- 必须优先通过 SOURCEMAP.map 理解模块职责并定位代码
- 必须遵守 @lib-zos/docs/guide/guide_zos.md 的编码风格
- 不允许发明未定义结构
- 实现必须与设计一致

--------------------------------------------------
【文档同步规则（必须）】

必须明确：

- 任何结构或逻辑变更，必须同步更新 docs/design/
- 必须同步更新 PROJECT_DESIGN.md 与 SOURCEMAP.map
- 如果代码与设计不一致，优先更新设计文档
- 不允许代码与设计长期不一致

--------------------------------------------------
【PROJECT_DESIGN.md 要求】

作为系统核心设计文档（SSOT）。

仅描述当前结构（as-is）。

必须包含：

- Overview
- Architecture
- Data Model
- Flow

禁止包含：

- TODO
- Roadmap
- 演进历史
- 版本信息
- 未来规划

--------------------------------------------------
【SOURCEMAP.map 要求】

使用表格格式：

| Module | Responsibility | Entry | Key Files | Path |
|--------|----------------|-------|-----------|------|

要求：

- Responsibility：一句话描述模块职责
- Entry：模块入口函数/入口文件
- Key Files：关键文件
- Path：代码路径
- 仅覆盖核心模块
- 内容必须简洁
- 不写设计说明
- 不写额外解释

--------------------------------------------------

面向开发者。

简要说明：

- 项目用途
- 如何使用
- Agent 驱动开发方式
- AGENTS.md 的作用
- docs/design/ 的作用
- 编码风格遵循 @lib-zos/docs/guide/guide_zos.md

--------------------------------------------------
【输出要求】

- 分文件输出
- 不要解释
- 不要额外建议
- 不要输出分析过程
- 不要输出教学内容
