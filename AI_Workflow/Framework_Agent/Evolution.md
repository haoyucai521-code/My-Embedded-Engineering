# Framework Agent 版本演进

记录 Framework Agent Prompt 从 1.0 到 3.0 的核心变化。

---

## 版本概述

| 版本 | 核心定位 | 生成文件 | 适用场景 |
|------|----------|----------|----------|
| 1.0 | 约束 Agent 行为 | AGENTS.md | 通用项目 |
| 2.0 | 建立轻量可执行约束体系 | AGENTS.md + PROJECT_DESIGN.md + SOURCEMAP.map + README.md | 通用项目 |
| 3.0 | 面向嵌入式工程的约束体系 | 同 2.0 | 嵌入式工程 |

---

## 1.0 → 2.0 提升

### 1. 从单文件到体系化

- 1.0 只生成 `AGENTS.md`。
- 2.0 要求生成 4 个文件，形成完整约束体系：
  - `AGENTS.md`：Agent 行为约束。
  - `docs/design/PROJECT_DESIGN.md`：核心设计事实来源。
  - `docs/design/SOURCEMAP.map`：模块定位索引。
  - `README.md`：面向开发者的项目说明。

### 2. 增加项目适配规则

- 区分"项目已有代码"和"项目为空"两种情况。
- 已有代码：基于现有结构总结，不改动架构。
- 空项目：仅定义最小合理结构，不做扩展设计。

### 3. 明确 Agent 行为边界

- 默认只读取当前任务相关文件。
- 默认禁止全仓扫描、无关重构、主动优化。
- 只有在怀疑跨模块、初始化流程、全局状态等问题时，才允许扩大分析范围。

### 4. 引入执行风格与探索预算

- 小步修改、最小可运行改动、一次只处理一个问题。
- 不输出教学内容、不输出未来规划、不重复需求。
- 扩大范围时必须说明原因，优先最小范围探索。

## 2.0 → 3.0 提升

### 1. 增加嵌入式特殊要求

3.0 最大的变化是加入嵌入式工程约束：

- 涉及 DMA、Cache、中断、时钟、引脚复用、功耗、Flash、BLE 事件链、RTOS 任务调度时，必须说明：
  1. 关键硬件约束。
  2. 潜在风险与触发条件。
  3. 规避方案。
- 未经验证的硬件行为必须标注"需验证"。
- 禁止编造寄存器地址、芯片特性、勘误信息。

### 2. 强化代码注释规范

- 代码注释使用中文。
- 注释重点说明：为什么这样做、边界条件、潜在风险、设计原因。
- 禁止无意义注释、重复逻辑注释、描述显而易见行为。

### 3. 文档同步规则更严格

- 任何结构或逻辑变化后，必须同步更新 `PROJECT_DESIGN.md` 和 `SOURCEMAP.map`。
- 代码与设计不一致时，优先更新设计文档。
- 禁止代码长期偏离设计。

### 4. 输出要求更贴合工程

- 保持与现有工程风格和目录结构一致。
- 分文件输出，不解释、不建议、不输出分析过程。

---

## 核心演进方向

1. **从约束到体系**：1.0 约束行为，2.0 建立完整约束体系，3.0 针对嵌入式领域深化。
2. **从通用到专用**：3.0 增加硬件相关约束，贴近嵌入式开发实际。
3. **从建议到强制**：文档同步、代码注释、事实来源等要求逐步明确和严格。
4. **从过程到结果**：输出要求逐步强调直接给结果，减少解释和教学。
